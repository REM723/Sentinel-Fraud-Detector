Process

How a transaction gets scored

Every request runs through the same pipeline that powers the live demo below.

01

Input normalization

Transaction type, amount, and origin/destination balances are validated and standardized using the same scaler from training.

02

Feature encoding

Transaction type is one-hot encoded across five categories: cash-in, cash-out, debit, payment, and transfer.

03

Model inference

A logistic regression classifier, weighted to handle class imbalance, produces a probability that the transaction is fraudulent.

04

Decision output

The result is returned as a fraud flag and a confidence score, ready to drive an approval, hold, or review action.

Under the hood

A focused, explainable model

No black boxes. The model relies on a small set of features that map directly to how money moves between accounts.

Algorithm

Logistic Regression with balanced class weighting, trained with a stratified train/test split on the PaySim transaction dataset.

Inputs

  • Transaction type
  • Transaction amount
  • Sender balance before and after
  • Receiver balance before and after

Why it works

Fraudulent transfers and cash-outs leave distinct fingerprints in balance changes, such as accounts being emptied entirely after a transfer. The model learns these patterns directly.
